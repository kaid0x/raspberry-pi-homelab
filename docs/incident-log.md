# Incident Log: DNS Deadlock & WAN Link Failure

Two separate real-world incidents that took the setup down, how they were diagnosed, the root causes, and the fixes. Documenting these because the *operations* side — diagnosing a live system under failure — is as important as the build, and the fixes made the setup meaningfully more resilient.

---

## Incident 1 — Self-hosted DNS deadlock (locked out remotely)

### Symptoms
- Whole-house internet intermittently failing; one room's Wi‑Fi down, another flaky.
- Could not SSH to the Pi over Tailscale from another network.
- Pi's lights were on, but it was unreachable.

### Diagnosis (via directly-attached monitor, since remote access was dead)
The Pi had booted fine — no crash, no SD corruption. On the console:
- `ping <router>` → worked (LAN was fine)
- `ping 8.8.8.8` → 100% loss (no internet)
- `tailscale status` → *"failed to resolve … no DNS fallback candidates remain"* and *"unable to connect to the coordination server."*
- `cat /etc/resolv.conf` → **generated by Tailscale**, pointing only at `100.100.100.100` (Tailscale's MagicDNS resolver).

### Root cause
A **circular dependency**. Tailscale had taken over the Pi's system DNS (`/etc/resolv.conf` → `100.100.100.100`). That resolver only works while Tailscale is connected. When the upstream internet dropped:
1. Tailscale couldn't reach its coordination server →
2. Tailscale's DNS (`100.100.100.100`) stopped resolving →
3. The Pi's *entire* system DNS depended on it, so the Pi could resolve **nothing** →
4. Tailscale couldn't get the DNS it needed to reconnect → **deadlock.**

The Pi's DNS server couldn't resolve its own dependencies once DNS was down. Classic self-hosted-DNS trap.

### Fix
1. Stop Tailscale from managing the Pi's DNS:
   ```
   sudo tailscale set --accept-dns=false
   ```
   (Client devices can still accept Tailscale DNS for Pi-hole-anywhere; only the *server* opts out — which is what breaks the deadlock.)
2. Give the Pi a resilient DNS order — its own Pi-hole first, public resolver as a guaranteed fallback:
   ```
   sudo nmcli connection modify <eth-conn> ipv4.dns "127.0.0.1 1.1.1.1"
   sudo nmcli connection modify <eth-conn> ipv4.ignore-auto-dns yes
   sudo nmcli connection down <eth-conn> && sudo nmcli connection up <eth-conn>
   ```
   Result — `/etc/resolv.conf`:
   ```
   nameserver 127.0.0.1
   nameserver 1.1.1.1
   ```

### Outcome
The Pi can now always resolve DNS regardless of Tailscale, Pi-hole/Unbound, or router state — so it can't get DNS-locked-out remotely again. Lesson: **an always-on server should never let a VPN client fully own its DNS with no fallback.**

## Incident 2 — WAN link failure (physical layer)

### Symptoms
- One router's network (the Pi's) had no internet; a different router on a separate path worked fine.
- Intermittent before going fully dead.

### Diagnosis (Pi used as a probe)
- `ping <router>` → 0% loss (Pi ↔ router perfect; router itself fine)
- `ping 8.8.8.8` → **66% loss** initially (degraded), then **100%** after cables were disturbed
- Router's **"Internet" indicator light was OFF** → no upstream WAN link at all.

Because the *other* router (different path) worked, this was isolated to **this router's uplink**, not an ISP/upstream outage.

### Root cause
A **failing/loose physical WAN connection**. The 66% packet loss *before anything was touched* proved the uplink cable/connector was already degrading. Reseating/replacing cables during troubleshooting temporarily worsened it (66% → 100%) before a solid cable + firm reseat restored a clean link.

### Fix
Replaced the WAN cable and reseated the upstream end firmly → "Internet" light came on solid → `ping 8.8.8.8` returned **0% loss**.

### Outcome / watch-list
Connection is clean now (0% loss, better than the pre-incident 66%). If flakiness recurs on this router specifically, suspect, in order: the WAN cable quality, the router's WAN port, or the aging router hardware itself.

## Diagnostic method that worked both times

1. **Isolate the scope** — "does the *other* network work?" instantly separated *upstream/ISP* problems (Incident 1) from *local single-router* problems (Incident 2).
2. **Use the Pi as a probe** — `ping <router>` vs `ping 8.8.8.8` pinpoints whether the break is LAN-side or WAN-side.
3. **Read the physical indicators** — the router's Internet light gave a definitive WAN-link status.
4. **Attach a monitor when the network is the very thing that's broken** — never rely solely on SSH to fix a networking/DNS failure.

## Resilience improvements adopted
- Pi no longer depends on Tailscale for its own DNS (`--accept-dns=false`).
- Pi has a public DNS fallback (`1.1.1.1`) after its own resolver.
- Reinforced the case for the still-outstanding **SD-card / config backup** (today was recoverable; SD corruption would not have been).
